# Use remote repository manifest.json for UUID extraction instead of parsing release assets

## Context

The index automation needs a stable UUID from each plugin so add-entry, mismatched-version, and scheduled refresh flows can detect identity drift. An earlier approach inspected the release asset archive to read `manifest.json`, but that introduced platform-specific tooling concerns and extra extraction logic.

## Content

Fetch `manifest.json` directly from the target repository on its default branch in JavaScript, and use that manifest as the UUID source of truth. The active release strategy still validates release naming and asset conventions, but UUID resolution now comes from the repository manifest rather than the packaged release artifact.

## Consequences / notes

- Workflows stay platform-neutral and do not rely on Python or archive parsing tools.
- UUID mismatches between `index.json` and the repository manifest are treated as a human-review condition.
- Repository layout now assumes `manifest.json` is available at the repository root on the default branch.
